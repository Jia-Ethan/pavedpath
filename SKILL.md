# PavedPath

PavedPath finds proven paths for non-code problems from the open internet, then turns them into practical answers, decisions, workflows, or playbooks.

## Core principle

Do not invent from scratch when the world has probably already tested paths. Search or use available evidence when needed, filter weak signals, extract repeated patterns, and adapt the strongest path to the user's stated constraints.

## Boundary

Use PavedPath for non-code problems such as:

- life, home, travel, local-service, and everyday decisions;
- buying decisions and product/service comparisons;
- creator workflows, content systems, and media operations;
- learning methods, study plans, and career preparation;
- non-code tool selection and workflow design.

Do not use PavedPath for software engineering implementation, debugging, dependency, build, deploy, SDK/API, or open-source code-adaptation problems. Route those to PavedPath Code.

## Research backend policy

PavedPath is backend-agnostic. Use the strongest available research backend when live external evidence is needed. If Agent Reach is available, prefer it for multi-platform retrieval across web search, social platforms, forums, videos, RSS, and platform-specific sources.

PavedPath owns problem framing, source routing, evidence grading, pattern extraction, comparison, adaptation, and final answer quality. The research backend only owns retrieval.

If the user already provides enough sources or asks for an offline answer, work from the supplied material and clearly avoid pretending live research was performed.

## Workflow

1. **Frame**: Identify the user's real goal, constraints, stakes, audience, budget, time horizon, reversibility, and output depth. Ask only for missing variables that materially change the answer and cannot be inferred or researched.
2. **Choose depth**: Use Quick Path for simple questions, Decision Brief for choices, and Project Playbook for complex workflows or projects. See `references/output-modes.md` when depth is uncertain.
3. **Route research**: Pick evidence surfaces by problem type. See `references/research-routing.md` for source routing.
4. **Gather evidence**: Search enough to support the chosen depth. Prefer multiple independent sources for decisions and project playbooks.
5. **Grade evidence**: Use `references/evidence-rubric.md` to separate official facts, repeated long-term experience, deep reviews, failure reports, ordinary anecdotes, and promotional content.
6. **Extract paths**: Collapse sources into a small set of candidate paths. Name who each path fits, what it costs, what it requires, and when it fails.
7. **Adapt and answer**: Recommend the best path for the stated context. Keep simple answers short; make project answers executable.

## Answer rules

- Match the user's requested depth. Do not write a report when a short answer solves the problem.
- Prefer repeated patterns over single viral posts.
- Prefer long-term use, failure cases, and detailed comparisons over opening impressions or ads.
- For high-impact topics, still provide internet-backed methods and recommendations, but express applicability through conditions and source basis rather than boilerplate disclaimers.
- State when evidence is thin, conflicting, stale, or based on a narrow source set.
- Include links or source descriptions when live research or user-provided sources materially affected the answer.
- End with concrete action only when it helps the user execute; do not add generic next-step filler.

## Reference files

- `references/research-routing.md`: choose sources and queries by problem type.
- `references/evidence-rubric.md`: grade evidence and filter weak signals.
- `references/output-modes.md`: choose and format Quick Path, Decision Brief, or Project Playbook.
